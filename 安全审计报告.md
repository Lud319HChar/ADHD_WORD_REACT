## 一、关键术语通俗解释

> 以下是我在审计报告中使用的专业术语，用白话解释一遍。

| 术语 | 白话解释 |
|------|----------|
| **XSS（跨站脚本攻击）** | 攻击者在网页中注入一段恶意 JavaScript，当其他用户打开这个页面时，恶意脚本就会在他们浏览器里执行。最典型的原因是开发者用 `innerHTML` 直接拼用户输入的内容，而没有做转义。 |
| **`innerHTML` vs `textContent`** | `innerHTML` 会把内容当作 HTML 解析（`<script>` 会被执行），`textContent` 只会当作纯文本（`<script>` 就是字面上的字符，不会执行）。**安全原则：除非你确定内容是安全的，否则永远用 `textContent`。** |
| **CORS（跨域资源共享）** | 浏览器的一种安全机制——默认禁止一个网站（例如 `evil.com`）用 JavaScript 去请求另一个网站（例如你的 `localhost:3000`）的数据。设置 `Access-Control-Allow-Origin: *` 等于**关掉了这个保护**，允许任何网站访问你的 API。 |
| **CSP（内容安全策略）** | 一种 HTTP 头部或 `<meta>` 标签，告诉浏览器"只允许从这些来源加载脚本/样式/图片"。这是防御 XSS 的最后一道防线。 |
| **路径遍历** | 攻击者通过在路径中使用 `../` 来访问服务器上不该被访问的文件。例如请求 `/download?file=../../etc/passwd`，如果服务器不做校验，攻击者就能读到系统密码文件。 |
| **监听 `0.0.0.0`** | 服务器绑定的地址。`127.0.0.1` = 只有本机能访问；`0.0.0.0` = 局域网内所有人都能访问。在公共 WiFi 下运行绑定了 `0.0.0.0` 的服务，你的 API 数据就会被同 WiFi 下的人看到。 |
| **供应链攻击** | 不直接攻击你的代码，而是攻击你依赖的第三方库。例如攻击者接管了某个 npm 包的维护权，在里面植入恶意代码，你的项目更新依赖时就中招了。 |
| **依赖幻觉** | AI（如 ChatGPT / Claude）在生成代码时可能会推荐一个**根本不存在**的第三方包名。攻击者可以提前用这个名字注册一个包，放入恶意代码，等不知情的开发者安装。 |
| **Prompt Injection（提示注入）** | AI 编程助手会读取你的整个项目代码作为上下文。如果某个文件里藏了一段恶意的"指令文本"（比如藏在注释或文档里），AI 可能会被这段指令误导，生成带漏洞或后门的代码。 |
| **硬编码凭据** | 把密码、API 密钥等敏感信息直接写死在代码里。一旦代码上传到 GitHub（即使是私有仓库），这些凭据就可能泄露。 |
| **`integrity` 属性** | HTML `<script>` 标签的一个属性，用来验证从 CDN 加载的 JS 文件是否被篡改过。浏览器会计算下载文件的哈希值，和 `integrity` 里的值比对，不匹配就拒绝执行。 |
| **ECB 加密模式** | AES 加密的一种模式，是最简单的模式——相同的明文块产生相同的密文块，所以密文会暴露出明文的图案（比如企鹅照片加密后还能看出企鹅的轮廓），已被公认不安全。 |
| **SAST / SCA** | SAST = 静态应用安全测试（扫描你的源代码找漏洞）；SCA = 软件组成分析（扫描你引用的第三方库是否有已知漏洞）。 |
| **Token / JWT** | Token 是一串用作身份凭证的字符串。JWT（JSON Web Token）是一种特定格式的 Token，包含三部分（头部、载荷、签名），常用来做 API 认证。`alg=none` 的意思是"不验证签名"，等于把门锁拆了。 |

## 二、项目当前风险清单

### 🔴 中风险（建议修复）

#### 1. CORS 配置全开

- **位置**：[server.js 第 46 行 / 第 54 行](file:///h%3A/Note/HTMLtest/server.js#L46)
- **问题代码**：

```javascript
res.setHeader('Access-Control-Allow-Origin', '*');
```

- **通俗解释**：这个设置的意思是"任何网站都可以用 JavaScript 跨域请求我的 API"。相当于你家门牌号公布在网上，谁来敲门都给开。
- **为什么是风险**：如果攻击者在局域网上架设一个钓鱼页面，它可以在你不知情的情况下读取你的训练数据、发指令给你的训练系统。
- **你的项目为什么这样写**：因为你的手机仪表盘需要跨域访问电脑上的 server。但正确的做法是只允许手机 IP 或固定域名，而不是 `*`。

---

#### 2. 文件路径遍历风险

- **位置**：[server.js 第 196-203 行](file:///h%3A/Note/HTMLtest/server.js#L196-L203)
- **问题代码**：

```javascript
let filePath = urlPath === '/' ? '/index.html' : urlPath;
filePath = path.join(__dirname, filePath);

if (fs.existsSync(filePath) && fs.statSync(filePath).isFile()) {
    serveStatic(res, filePath);
}
```

- **通俗解释**：攻击者可以通过 `http://localhost:3000/../../../Windows/System32/drivers/etc/hosts` 这样的 URL，尝试读取服务器上的任意文件。
- **为什么危险**：`fs.existsSync` 和 `fs.statSync` 会如实检查攻击者构造的路径，如果路径存在且是文件，就会返回文件内容。
- **修复方案**：在读取文件之前，先用 `path.resolve` 把路径解析成绝对路径，然后检查它是否在你允许的目录范围内。

---

#### 3. API 无任何身份认证

- **位置**：[server.js 第 53-171 行](file:///h%3A/Note/HTMLtest/server.js#L53-L171)
- **通俗解释**：所有 `/api/progress`、`/api/stats`、`/api/command` 都不需要任何密码或 Token 就能访问。局域网内任何人都可以：
  - 查看你的训练进度
  - 修改你的不熟词库
  - 给训练系统发指令
- **为什么是风险**：你用 `0.0.0.0` 监听，如果你连到咖啡店的公共 WiFi，同网络下的任何人都可以用你电脑的 IP 访问这些 API。

---

#### 4. 服务监听在所有网络接口

- **位置**：[server.js 第 208 行](file:///h%3A/Note/HTMLtest/server.js#L208)
- **问题代码**：

```javascript
server.listen(PORT, '0.0.0.0', () => {...});
```

- **通俗解释**：`0.0.0.0` 的意思是"监听本机所有网络接口"，这样局域网内的手机才能访问。但这也意味着**任何能连到你电脑网络的人**都能访问。
- **建议**：如果只在家庭局域网内使用，且路由器有防火墙，风险不大。但绝对不要在公共 WiFi 下运行。

#### 6. 缺少 CSP（内容安全策略）

- **问题**：所有 HTML 页面都没有设置 Content-Security-Policy。
- **通俗解释**：CSP 是浏览器的一道安全防线——你告诉浏览器只信任哪些来源的脚本/样式。即使有不安全的 `innerHTML` 或 XSS 注入，CSP 可以作为最后一道防线阻止恶意脚本执行。
- **为什么没有**：Vibe Coding 几乎不会主动添加 CSP，因为在快速原型阶段这不是"功能需求"。

#### 7. CDN 依赖无完整性校验

- **位置**：[index.html 第 8 行](file:///h%3A/Note/HTMLtest/index.html#L8) / [mobile-dashboard.html 第 7 行](file:///h%3A/Note/HTMLtest/mobile-dashboard.html#L7)
- **问题代码**：

```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.5/gsap.min.js"></script>
```

- **通俗解释**：你从 cdnjs 加载 GSAP 库，但没有 `integrity` 属性来验证文件是否被篡改。如果 CDN 被攻击（或你连接了恶意 WiFi 热点，DNS 被劫持），浏览器加载的 GSAP 文件可能含有恶意代码。
- **修复示例**：

```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.5/gsap.min.js"
        integrity="sha384-..." crossorigin="anonymous"></script>
```

## 三、AI 生成代码 / Vibe Coding 的系统性风险

> 以下是根据安全研究文献和行业报告整理的 Vibe Coding 常见风险类型。
> 我在你的项目代码中已标注（★）表示该项目**实际存在**此特征。

### 风险 1：依赖幻觉 ★（本项目暂无，但最具破坏力）

这是 Vibe Coding **独有的**风险——传统开发中几乎不存在。

| 子类型 | 通俗解释 |
|--------|----------|
| **虚构包名** | AI 推荐一个不存在的 npm/pip 包名。攻击者抢先注册同名包，放入恶意代码。这被称为「slopsquatting」攻击。 |
| **拼写变体劫持** | `requets` 是 `requests` 的拼写错误，但 AI 会当真。攻击者注册所有常见拼写错误的包名。 |
| **过期版本** | AI 训练数据截止于某个日期，推荐的是旧版本（可能已被发现有严重漏洞） |
| **过度引入依赖** | AI 倾向于为一个简单功能引入整个重型库（如为单行函数引入 `lodash`），增加了攻击面 |

---

### 风险 2：XSS 模式 ★（本项目大量存在）

这是 AI 生成前端代码中**最普遍的漏洞类型**。

```javascript
// ★ 你的代码中大量使用的模式（来自 wordDB 等静态数据，当前安全）：
element.innerHTML = `<div>${data.field}</div>`;

// 正确的写法应该是：
element.textContent = data.field;
// 或者先用 escapeHtml() 转义
```

**为什么 AI 总生成 `innerHTML`？**

因为 Stack Overflow 和博客教程中大量使用 `innerHTML`（方便、直观、代码少），AI 在学习时就认为这是"标准写法"。

---

### 风险 3：CORS 配置过于宽松 ★（本项目存在）

```javascript
// ★ 你的 server.js 第 46 行就是这样写的：
res.setHeader('Access-Control-Allow-Origin', '*');
```

AI 在处理跨域需求时，最简单的"能跑就行"的方案就是设为 `*`。

---

### 风险 4：硬编码凭据（本项目无）

AI 生成代码时可能写出类似这样的内容：

```javascript
// 典型 AI 生成模式：
const API_KEY = "sk-your-key-here";  // 直接写在代码里
const DB_PASSWORD = "admin123";
const JWT_SECRET = "mysecret";
```

---

### 风险 5：弱密码学（本项目 N/A）

```python
# AI 常见推荐：
hashlib.md5(password)      # MD5 已被破解
random.randint(0, 999999)  # 不是密码学安全的随机数
AES.new(key, AES.MODE_ECB) # ECB 模式不安全
```

---

### 风险 6：缺少认证与授权 ★（本项目存在）

```javascript
// ★ 你的 server.js：任何 API 端点都不需要认证
if (urlPath === '/api/progress' && method === 'GET') {
    // 直接返回数据，不验证请求者身份
}
```

AI 在生成快速原型时天然倾向于跳过认证，因为认证不是核心功能。

---

### 风险 7：路径遍历 ★（本项目存在风险）

```javascript
// ★ 你的 server.js 第 196-203 行
// 直接把用户 URL 拼接到文件路径
filePath = path.join(__dirname, filePath);
```

攻击者在 URL 中使用 `../` 可以跳出项目目录。

---

### 风险 8：Prompt Injection（间接攻击）

这不是代码层面的漏洞，而是**开发流程中**的风险：

| 场景 | 说明 |
|------|------|
| **恶意注释** | 合作者在你项目的注释中写入隐藏指令，AI 读取整个项目后，可能被这些指令影响 |
| **恶意文档** | 第三方库的 README 中包含 prompt 注入文本，AI 在分析该库时被误导 |
| **上下文截断** | 对话很长后，最初要求的安全约束（如"用参数化查询"）会被 AI 遗忘 |

---

### 风险 9：Vibe Coding 特有的结构性问题

| 问题 | 通俗解释 |
|------|----------|
| **"能跑就行"陷阱** | 代码功能正常 → 开发者不会再审计 → 安全隐患遗留下来。这是 Vibe Coding 最大的方法论风险。 |
| **新手学习不安全模式** | 如果初学者通过 AI 学编程，会把 `innerHTML` 当作默认做法学会 |
| **安全代码的幻觉** | AI 可能编造一个不存在的安全函数名（如 `sanitizeUser()`），但函数体是空的 |
| **上下文遗忘** | 你在第 1 条消息里说"请做输入校验"，AI 在第 20 轮就忘了 |

---

### 风险总结矩阵

| 风险类别 | 严重性 | AI 代码中普遍性 | 你的项目是否存在 | 检测难度 |
|----------|:------:|:---------------:|:----------------:|:--------:|

| CORS 全开 | 🟠 高 | 高 | ★ **存在** | 低 |
| 硬编码凭据 | 🔴 严重 | 高 | ✅ 无 | 低 |
| 弱加密 | 🟠 高 | 高 | ✅ N/A | 中 |
| 缺少认证 | 🔴 严重 | 高 | ★ **存在** | 高 |
| 路径遍历 | 🟠 高 | 中 | ★ **存在风险** | 低 |
| 业务逻辑缺陷 | 🟡 中 | 高 | ✅ 当前简单无影响 | 极高 |

#### 修复 1：限制 CORS

**文件**：[server.js](file:///h%3A/Note/HTMLtest/server.js)

```javascript
function setCORS(res, req) {
    const origin = req.headers.origin || '';
    // 允许本地和局域网访问
    if (origin.startsWith('http://localhost') || 
        origin.startsWith('http://192.168.') ||
        origin.startsWith('http://127.0.0.1') ||
        origin.startsWith('http://10.') ||
        origin.startsWith('http://172.')) {
        res.setHeader('Access-Control-Allow-Origin', origin);
    }
}
```

## 五、Vibe Coding 安全自查清单

> 如果你以后继续用 Vibe Coding 方式开发，每次生成新功能后，走一遍这个清单：

### 代码层面

- [ ] 有没有用 `innerHTML`？如果有，数据来源是静态的还是用户输入的？
- [ ] 有没有引入新的第三方包？在 npmjs.com 上确认过包名确实存在吗？

## 总结

你的 CET6 项目是一个本地学习工具：
- **没有收集用户隐私**（好）
- **没有使用数据库，避免了 SQL 注入**（好）
- **没有硬编码密钥**（好）
- **当前最大的风险是 server.js 的网络暴露面**：CORS 全开 + 无认证 + 监听 `0.0.0.0`
- **代码中的 `innerHTML` 是 Vibe Coding 的典型产出**，当前安全但将来如果数据源变化就会有 XSS 风险

> **一句话**：项目整体安全水平是好的，上述问题都不紧急，但如果项目将来要部署到公网或多人使用，需要先修复标注为 🔴 的项目。
